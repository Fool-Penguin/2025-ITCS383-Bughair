## 2. Deliverables (D1 – D5)

### 📦 D1: Working Prototype & CI/CD Pipeline
- [ ] Deploy the backend on a **free cloud service** (e.g. Render, Vercel).
- [ ] Set up **GitHub Actions** for automated builds & tests.
- [ ] Integrate **SonarQube Cloud** to check code quality on every push.
- [ ] Have a **working prototype** ready for both **web** and **mobile**.

### 📊 D2: Code Quality Report → `D2_CODE_QUALITY.md`
- [ ] Create the file `D2_CODE_QUALITY.md`.
- [ ] Show **SonarQube results** — compare **before vs. after** your changes.
- [ ] New code must have **> 90% test coverage**.

### 📝 D3: Change Request Analysis → `D3_CHANGE_REQUESTS.md`
- [ ] Create the file `D3_CHANGE_REQUESTS.md`.
- [ ] Break down the 3 new features into **at least 8 Change Requests (CRs)**.
- [ ] Categorize them exactly as:
  - **2 Corrective** (fixing bugs)
  - **2 Adaptive** (adjusting to new environment/platform)
  - **2 Perfective** (improving features/performance)
  - **2 Preventive** (preventing future problems)
- [ ] Use the **CR table format** from the project description.

### 🔍 D4: Impact Analysis → `D4_IMPACT_ANALYSIS.md`
- [ ] Create the file `D4_IMPACT_ANALYSIS.md`.
- [ ] Draw a **traceability graph** of the whole software.
- [ ] Highlight the **affected parts** in a separate version of the graph.
- [ ] Create a **directed graph of SLOs** (Software Lifecycle Objects = code modules).
- [ ] Build a **connectivity matrix** showing distances between SLOs.
- [ ] Write an **analysis** explaining:
  - Which CRs were **easy** and which were **hard** — and why.
  - What you'd **expect from the previous developers** (e.g. better docs, cleaner code, etc.).

### 🤖 D5: AI Usage Report → `D5_AI-USAGE.md`
- [ ] Create the file `D5_AI-USAGE.md`.
- [ ] Document **every activity** where AI tools were used to help with the project.
